# Java Spring IoC（控制反转）

## 为什么要先学 IoC

前面 Part 1 讲完了注解、反射、Lambda，这三者是 Spring 的语言层基础设施。接下来进入 Spring 的核心哲学 -- 而一切的起点就是 IoC（Inversion of Control，控制反转）。

IoC 不是 Spring 发明的，它是一种设计原则。但 Spring 把这个原则做到了极致，成为了整个框架的基石。如果 IoC 理解不透彻，后面学 Bean、DI、容器都会感觉"知道怎么用但不知道为什么这么用"。

---

## 核心概念

### IoC 是什么

**IoC（Inversion of Control，控制反转）是一种设计原则：将"对象创建和管理依赖"的控制权从程序自身反转给外部容器。**

传统编程是你自己 `new` 对象、自己管理对象之间的关系。IoC 模式下，你只需要声明"我需要什么"，容器替你完成对象的创建和依赖的组装。

一个直观的类比：想象你去餐厅吃饭。传统方式是你自己走进厨房找食材做菜，IoC 方式是你只需要告诉服务员"我要一份牛排"，厨房负责采购、烹饪、装盘，最后端给你。关键反转是：不是你主动去获取依赖，而是依赖被"端到"你面前。

### 为什么需要 IoC

在复杂应用中，对象之间的依赖关系形成一张网。手动管理这张网（创建顺序、生命周期、依赖组装）极其繁琐且容易出错。IoC 把这些"脏活累活"交给容器，让业务代码只关注业务逻辑。

### 没有 IoC 会怎样

你需要手动 `new` 每个对象、手动管理创建顺序、手动传递依赖。组件之间硬编码耦合，换一个实现要改遍调用方。单元测试变成噩梦 -- Mock 一个对象需要手动构造整个依赖树。有了 IoC，这些问题由容器统一解决，代码变得松耦合、可测试、可替换。

## 概念深入解释

### 控制权到底"反转"了什么

很多人对"反转"这个词感到困惑。具体来说，以下控制权从程序代码转移到了容器：

| 控制权 | 传统方式 | IoC 方式 |
|--------|----------|----------|
| 创建时机 | 你在代码中 `new` 决定 | 容器在合适的时机创建 |
| 生命周期 | 你手动销毁或靠 GC | 容器管理创建、初始化、销毁 |
| 依赖组装 | 你手动调用 `setXxx()` | 容器自动注入 |
| 实现选择 | 你硬编码具体类 | 容器根据配置选择实现 |
| 作用域 | 你自己控制单例/多例 | 容器按配置管理作用域 |

当所有控制权都交给容器后，你的代码变成了纯粹的"声明" -- 声明自己是什么（`@Service`）、声明自己需要什么（`@Autowired`）。

### 为什么手动管理依赖不可维护

考虑一个简单的依赖链：`Controller` 依赖 `Service`，`Service` 依赖 `Repository`，`Repository` 依赖 `DataSource`。在传统方式下：

```java
Repository repository = new Repository(DataSource.getConnection());
Service service = new Service(repository);
Controller controller = new Controller(service);
```

这只是 3 层依赖。实际项目可能有几十上百个组件，每个组件可能有多个依赖。如果全部手动管理：创建顺序敏感、重复代码泛滥、紧耦合、难以测试。IoC 让容器接管这一切，你只需要声明关系。

### IoC 的几种实现方式

IoC 是原则，不是具体技术。实现 IoC 有多种方式：

- **依赖注入（DI）** -- 最常见的方式。容器把依赖"注入"到需要它的对象中。Spring 的核心机制。
- **依赖查找（Dependency Lookup）** -- 对象主动向容器索取依赖（`context.getBean()`）。Spring 支持但不推荐。
- **Service Locator 模式** -- 通过全局注册表查找服务。依赖查找的变体。

Spring 选择依赖注入作为主要方式，因为它对代码的侵入性最小 -- 你的类仍然是 POJO，不需要继承 Spring 的类或实现 Spring 的接口。

### IoC 容器架构

```
                    ┌─────────────────────────────────────┐
                    │         BeanFactory (根接口)          │
                    │   - getBean() / containsBean()      │
                    └──────────────┬──────────────────────┘
                                   │ 继承
                    ┌──────────────▼──────────────────────┐
                    │      ApplicationContext (常用)        │
                    │   - 国际化 (MessageSource)            │
                    │   - 事件发布 (ApplicationEventPublisher)│
                    │   - 资源加载 (ResourceLoader)          │
                    └──────────────┬──────────────────────┘
                                   │ 实现
                 ┌─────────────────┼─────────────────┐
                 ▼                 ▼                 ▼
    AnnotationConfigAC    ClassPathXmlAC    WebApplicationContext
    (注解驱动，主流)       (XML，遗留项目)     (Web 环境)
```

## 核心要点

1. **IoC 是设计原则，不是具体技术。** 它解决"谁负责创建和管理对象依赖"的问题。
2. **控制反转 = 将对象创建和依赖管理的控制权从代码交给容器。**
3. **DI（依赖注入）是 IoC 最主流的具体实现方式。** Spring 选择 DI 是因为它对代码侵入性最小。
4. **IoC 的核心价值是解耦。** 让组件只关心"我需要什么接口"，不关心"谁来实现、怎么创建"。
5. **Spring IoC 容器是 IoC 的物理实体。** 它是整个 Spring 框架的运转中心。

---

## 常见误区

- **以为 IoC 就是依赖注入。** IoC 是原则，DI 是实现。依赖查找也是 IoC 的实现方式。
- **以为用了 Spring 就是用了 IoC。** 如果你在 Spring 环境中手动 `new` 对象、手动传递依赖，那你就没有真正利用 IoC。IoC 要求你把对象生命周期全部交给容器。
- **把 IoC 理解成"代码变得不可控"。** 恰恰相反，是把复杂的控制逻辑集中交给经过验证的容器，反而更可控、更不易出错。
- **以为 IoC 让代码变慢。** Spring 的 IoC 开销几乎全部集中在启动阶段。运行时 Bean 已经创建好、依赖已经注入完毕，没有额外开销。
