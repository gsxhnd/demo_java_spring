# Java Spring 通信协议选型

## 为什么要学通信协议选型

前面三节我们分别学了 WebSocket、MQTT、Modbus 三种通信协议。加上已经熟悉的 HTTP/REST，现在你手上至少有了四种协议可以选择。问题来了：面对一个具体的业务需求，选哪个协议？选错了，可能只是"用起来别扭"；也可能导致性能瓶颈、架构复杂度飙升、甚至根本上不了线。

协议选型不是"哪个协议技术更先进就用哪个"，而是**在特定约束下选择最合适的工具**。这节从场景出发，帮你建立选型直觉。

## 核心概念

### 通信协议选型是什么

通信协议选型是根据应用场景的约束条件（客户端类型、网络环境、数据模式、可靠性要求、团队熟悉度等），在可用协议中选择最合适通信方式的过程。

**换个说法：** 选协议就像选交通工具。去隔壁城市（低延迟、短距离）开车就够了；跨越大洋（长距离、大容量）需要轮船；要在深山老林里（恶劣环境、无基础设施）可能是骡马。工具没有绝对好坏，只有适不适合场景。

### 为什么需要系统性选型

**痛点场景：** 团队习惯用 HTTP 就所有场景都用 HTTP。某天接了一个 IoT 项目，传感器每 5 秒上报一次数据，5000 台设备。发现 HTTP 短连接开销太大，临时换成 MQTT — 但架构已经围绕 HTTP 做了鉴权、限流、日志、监控，换成 MQTT 意味着基础设施全部重做。

**设计动机：** 在项目启动前，花 1 小时做协议选型，可能避免 3 个月后的架构返工。

### 没有系统性选型会怎样

**困境：** 用习惯的技术硬套所有场景。好处是团队熟悉，坏处是当场景不匹配时，维护成本持续攀升，最终可能变成"遗留系统"。

**有了选型框架之后：** 用一组结构化的维度来评估，而不是凭感觉。选型变成可以讨论、可以复盘的过程。

## 概念深入解释

### 协议对比总览

| 维度 | HTTP/REST | gRPC | WebSocket | MQTT | Modbus |
|------|-----------|------|-----------|------|--------|
| 通信模式 | 请求-响应 | 请求-响应 + 流 | 全双工 | 发布-订阅 | 主-从轮询 |
| 消息方向 | 客户端主动 | 客户端主动 + 服务端流 | 双向 | 任意方向（通过 Broker） | Master 主动 |
| 传输层 | TCP (HTTP/1.1 或 HTTP/2) | TCP (HTTP/2) | TCP (升级自 HTTP) | TCP | 串行 (RTU) 或 TCP |
| 消息格式 | JSON / XML / 任意 | Protobuf（二进制） | 文本/二进制 | 二进制（极小） | 二进制（极简） |
| 头部开销 | 大（HTTP headers） | 中（HTTP/2 + Protobuf） | 小（升级后） | 极小（2 字节起） | 极简 |
| 连接模式 | 短连接（可 Keep-Alive） | 长连接 | 长连接 | 长连接 | 按需连接 |
| 浏览器支持 | 原生 | 需 grpc-web | 原生 | 需 MQTT over WS | 不支持 |
| 强类型契约 | OpenAPI / 手动 | .proto 自动生成 | 自定义 | 自定义 | 地址表 |
| 流式传输 | SSE（单向） | 双向流 | 双向 | 实时推送 | 轮询 |
| 安全 | TLS + OAuth/JWT | TLS + Token | TLS + Token（握手） | TLS + 用户名密码 | 无内置 |

### 按场景选型

**场景一：通用 Web API（增删改查）**

> 前端/移动端通过 RESTful API 进行标准的数据操作

**推荐：HTTP/REST**

理由：浏览器原生支持，生态最成熟（OpenAPI 文档、Postman 测试、负载均衡器支持、CDN 缓存可用的 GET 请求）。学习成本最低，开发效率最高。除非有明确的性能瓶颈，否则 REST 是默认选择。

---

**场景二：微服务内部高性能通信**

> 订单服务需要频繁调用库存服务、用户服务，对延迟和吞吐量有高要求

**推荐：gRPC**

理由：基于 HTTP/2 的多路复用和二进制 Protobuf 编码，延迟和带宽显著优于 REST + JSON。`.proto` 文件作为强类型契约，避免了 REST 中"接口文档和代码不一致"的问题。支持双向流（适合文件传输、实时数据同步）。

---

**场景三：浏览器实时推送（聊天、通知、协作）**

> 网页上需要实时展示聊天消息、系统通知、协作状态变化

**推荐：WebSocket + STOMP**

理由：浏览器原生支持，全双工双向通信，服务端可主动推送。STOMP 提供发布/订阅模型，开发体验接近写 Controller。配合 SockJS 可降级到 HTTP 方案。

---

**场景四：IoT 设备数据采集**

> 成千上万个低功耗传感器需要定期上报数据，网络环境不稳定，设备资源有限

**推荐：MQTT**

理由：极低带宽开销、QoS 分级、遗嘱消息、离线消息保留 — 这些是为 IoT 量身定做的。通过 Broker 解耦设备和后端，支持百万级连接。

---

**场景五：工业设备直连控制**

> 需要直接读取 PLC、变频器、仪表的寄存器数据，设备只有 RS-485 串口

**推荐：Modbus RTU/TCP**

理由：工业设备事实标准，几乎 100% 的 PLC 和工业仪表都支持 Modbus。不需要设备支持 TCP/IP，RS-485 即可通信。适合本地设备控制环路。

---

**场景六：文件传输 / 大消息**

> 需要上传大文件或发送大块数据

**推荐：HTTP Multipart（小文件）/ gRPC 流（大文件或多文件）**

理由：HTTP 分块传输成熟可靠，CDN 支持好。gRPC 的流式传输适合持续的数据流。WebSocket 也可传输二进制数据，但对于一次性的大文件没有优势。

---

### 协议组合使用

在实际项目中，一个系统可能同时使用多种协议：

```
移动 App ── HTTP/REST ──┐
                        ├── Spring 微服务 ── gRPC ── 内部服务
浏览器    ── WebSocket ─┘        │
                                │
传感器    ── MQTT ── Broker ────┘
                                │
PLC/仪表  ── Modbus ── 网关 ────┘
```

例如：传感器通过 MQTT 上报数据到 Broker，Broker 将数据桥接到 Kafka，Spring 服务消费 Kafka 消息做分析，分析结果通过 WebSocket 推送到浏览器，同时提供 HTTP API 供移动端查询历史数据。PLC 设备通过 Modbus 连接到边缘网关，网关将数据转为 MQTT 消息发送到 Broker。

### 选型决策流程图

```
需要浏览器直连？
├── 是 → 请求-响应模式？
│        ├── 是 → HTTP/REST
│        └── 否（实时推送） → WebSocket
│
└── 否 → 微服务间调用？
         ├── 是 → 高性能要求？
         │        ├── 是 → gRPC
         │        └── 否 → HTTP/REST
         │
         └── 否 → IoT 设备？
                  ├── 是 → 低功耗/低带宽/大量设备？
                  │        ├── 是 → MQTT
                  │        └── 否 → 考虑 HTTP
                  │
                  └── 否 → 工业设备/PLC？
                           ├── 是 → Modbus RTU/TCP
                           └── 否 → 回到 HTTP/REST（默认）
```

### 反模式：协议万能论

最常见的选型反模式是"XX 协议是最好的，所有场景都用它"：

- 所有场景用 gRPC → 浏览器需要 grpc-web 转码，增加维护成本。简单 CRUD 场景 Protobuf 定义过于繁琐。
- 所有场景用 WebSocket → 对于请求-响应模式，WebSocket 增加连接管理复杂度，没有收益。REST 更适合这种场景。
- 所有场景用 MQTT → 开发团队需要学习一套全新的 Broker 运维体系，对于只有 5 个设备的小项目杀鸡用牛刀。

## 核心要点

1. **协议选型不是选技术最先进的，而是选最匹配场景约束的。**
2. **HTTP/REST 是默认选择：** 没有明确理由换协议时，用 REST 不会错。
3. **浏览器是硬约束：** 浏览器原生只支持 HTTP 和 WebSocket，gRPC 需要转码，MQTT 需要 over WebSocket。
4. **协议可以组合使用：** 一个系统同时用 HTTP（API）+ WebSocket（推送）+ MQTT（设备）+ gRPC（服务间）是合理的，关键是边界清晰。
5. **选型考虑团队能力：** 引入一个新协议意味着运维、监控、日志、安全全链路需要适配，组织成本也是约束条件。

## 常见误区

- **API 设计用 POST 处理所有操作，把 REST 用成了 RPC。** `POST /getUser`、`POST /deleteOrder` — 没有利用 HTTP 方法（GET/POST/PUT/DELETE）和资源 URL 的表达力。REST 的风格约定（非强制）能让 API 自解释，降低理解成本。
- **WebSocket 连接建立后不做心跳检测。** 代理服务器（Nginx、负载均衡器）可能静默关闭空闲连接。没有心跳（Ping/Pong）的 WebSocket 连接会在几分钟内断掉而不被双方感知。STOMP 协议内置了心跳机制，记得配置。
- **gRPC 忽略了负载均衡。** HTTP/2 长连接模式下，传统 L4 负载均衡（每次 TCP 连接才均衡）不适用。gRPC 需要 L7 负载均衡（如 Linkerd、Envoy）或客户端负载均衡（配合服务发现）。
- **MQTT Topic 设计成和 HTTP URL 一样长。** MQTT 的 Topic 字符串是每一条消息都携带的，过长的 Topic 会显著增加带宽开销。保持 Topic 简洁，用层级表达语义即可。
- **在一个系统中混用多种兼容协议导致运维爆炸。** 每多一种协议，日志格式、监控指标、安全策略、中间件运维都要多一份配置。在满足需求的前提下，协议种类越少越好。不要因为"MQTT 看起来有趣"就在没有 IoT 设备的项目里引入它。
- **Modbus 的字节序和 MQTT/HTTP 的数据格式不一致导致数据解读错误。** Modbus 是 Big-Endian 的 16-bit 寄存器，MQTT/HTTP 的 JSON 按平台字节序。在协议转换时必须根据设备手册做显式的字节序转换和数据类型映射。
